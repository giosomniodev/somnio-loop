# do

## Descripción general

El skill `do` es el punto de entrada principal del plugin somnio-loop. Acepta un ticket o descripción de tarea y orquesta el loop agéntico completo de punta a punta — desde el triage y la planificación hasta la implementación, la verificación y la entrega del artefacto final.

## Cuándo usarlo

- El usuario pega un ticket de Jira/Linear/GitHub o una descripción de tarea en texto libre
- El usuario dice `/loop:do`, "corré esto de punta a punta" o "ejecutá el flujo completo"
- Cualquier solicitud que implique un entregable de múltiples fases delegado a ejecución autónoma

NO usar para búsquedas de un solo paso o ediciones rápidas — esas se ejecutan directamente sin el overhead del loop.

## Entradas

| Entrada | Descripción |
|---|---|
| Texto del ticket | Cuerpo completo del ticket, lista de bullets, o ID de ticket bare (ej. `TKT-123`) |
| `.loop/config.yaml` | Preset de autonomía y configuración de compuertas |
| `.loop/state.md` | Contexto de corridas previas y watch list |
| `--autonomy=<preset>` | Override por corrida opcional al final del ticket |

## Salidas

| Salida | Ubicación |
|---|---|
| Entregable(s) final(es) | Carpeta de outputs del usuario |
| `run-report.md` | Junto a cada artefacto |
| `plan.yml` | `.loop/plans/<ts>-<ticket_hash>.yml` |
| `trace.json` | `.loop/traces/<ts>-<ticket_hash>.json` |
| Run-log actualizado | `.loop/run-log.md` |

El mensaje final en el chat siempre incluye un resumen ejecutivo y una tabla de consumo por agente (modelo, invocaciones, tiempo, tokens, costo).

## Definición de Done

Una corrida no está completa hasta que **todos** los elementos siguientes estén presentes. El orquestador verifica este checklist antes de entregar el mensaje final — los ítems faltantes disparan un re-despacho o un estado `partial` explícito, nunca un skip silencioso.

| Ítem | Requerido |
|---|---|
| Cada artefacto en `final_artifacts:` existe y no está vacío | Siempre |
| `verify-report.md` con una línea `BLOCKING:` | Siempre |
| `run-report.md` con status, fases, tokens, costo, lista de artefactos | Siempre |
| Tabla de consumo por agente con totales consistentes | Siempre |
| Entrada en `run-log.md` actualizada al status final | Siempre |
| `trace.json` guardado en `.loop/traces/` | Siempre |
| `plan.yml` archivado en `.loop/plans/` | Siempre |
| Status del ticket actualizado / PR creado / notificación enviada | Cuando está configurado |

Si la corrida debe entregarse como `partial` (presupuesto agotado, máximo de reintentos alcanzado, findings del verifier irresolubles), el orquestador muestra exactamente qué está presente y qué falta, y agrega los ítems faltantes a `state.md → High Priority` para la siguiente corrida.

## Multi-ticket: un branch y un PR por ticket

Cuando corrés múltiples tickets (`do "MH-100 MH-101"` o `--parallel`), cada ticket está **completamente aislado**. Este es el error más común en modo multi-ticket: workers de dos tickets distintos terminando en el mismo branch y generando un PR que mezcla cambios no relacionados.

**El invariante:** un ticket → un branch → un PR. Siempre. Sin excepción.

| Scope | Aislado por ticket |
|---|---|
| Branch | `feature/MH-100-...` y `feature/MH-101-...` — creados independientemente desde el mismo base |
| Worktrees | Cada ticket tiene sus propios directorios `wt-<ticket_id>-<worker_n>` |
| Directorio de output | `/tmp/loop-runs/<ts>-MH-100/` y `/tmp/loop-runs/<ts>-MH-101/` |
| Entrada en run-log | Una entrada por ticket |
| Creación de PR | La Phase 7 corre independientemente por ticket — NO una sola vez para el batch |

### Cómo funciona el despacho paralelo

Cuando se usa `--parallel`, cada ticket se despacha como un **sub-agente independiente** que corre el protocolo completo de 8 fases de `do` por su cuenta. El orquestador padre NO corre fases ni crea PRs — solo recolecta resultados y arma la tabla resumen.

```
do (orquestador padre)
  ├─ Agent: ticket MH-100 → loop completo → branch feature/MH-100-... → PR #444
  ├─ Agent: ticket MH-101 → loop completo → branch feature/MH-101-... → PR #445
  └─ tabla resumen (armada por el padre después de que ambos retornan)
```

Al final, el orquestador muestra una tabla consolidada con un PR URL por ticket:

```
| Ticket | Status      | Branch                    | PR   |
|--------|-------------|---------------------------|------|
| MH-100 | ✅ completed | feature/MH-100-add-cache  | #444 |
| MH-101 | ✅ completed | feature/MH-101-fix-login  | #445 |
```

## Ejemplo

```bash
/loop:do TKT-421 — Agregar rate limiting a la API pública

Criterios de aceptación:
- 429 devuelto después de 100 req/min por API key
- Headers X-RateLimit-* en cada respuesta
- Los tests de integración existentes siguen pasando
```
